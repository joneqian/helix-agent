# 9.5 分布式任务队列 — 设计（2026-06-16）

> 能力项 9.5「托管任务队列 / 分布式任务调度」★2→★5。范围 **B**（用户拍板）：
> 异步 run 队列 + 加固现有后台 worker 多副本安全。house 风格（Postgres-first，零 Celery/broker）。

## 1. 现状（核代码）

- **run 全同步 SSE 进程内执行**：`POST /v1/sessions/{thread}/runs`（`runs.py:trigger_run`）建 `RunRecord`
  → 立即 `asyncio.create_task(run_agent(...))` 在**当前进程**跑 → `StreamingResponse` SSE 流式。`PENDING`
  是瞬态（create→立即 RUNNING），**无队列、无跨进程分发**。
- **9.4 已留的接力件**：agent_run 租约（`claimed_by`/`lease_until`/`heartbeat_at`/`reclaim_count`）+
  `RunStore.reclaim` CAS + `OrphanSweep._respawn`（adopt + `run_agent(graph_input=None)` 从 checkpoint 续）。
- **输入未持久化**：`graph_input` 在 POST handler 现造（`payload.input`/`image_refs`/`untrusted_content`
  + `built.*`），agent_run **不存 input**。queued run 要在别的进程执行,必须先持久化用户输入。
- **现有 worker 多副本不安全**（潜伏 bug）：`EvalWorker.run_once`（`list QUEUED → set_status(RUNNING)`，无
  CAS）/ `TriggerScheduler._fire_due_cron`（无 CAS）/ approval sweep / reaper——蓝绿两实例都扫到→**重复执行**。
  仅 `OrphanSweep` 安全（reclaim CAS）。app.py lifespan **无条件**起这些 worker（只 enable 闸，无单点选举）。
- **SSE 解耦已就绪**：`GET /runs/{id}/events` 终态走 `RunEventStore.list` durable replay / live 走
  `bridge.subscribe`；StreamBridge per-process 内存。客户端可经 events 端点重连到「别处执行的 run」（durable
  replay），无需持 SSE 原连接。

## 2. 设计

### 2.1 Phase 1 — 异步 run 队列

**新增 `RunStatus.QUEUED`**（agent_run CHECK 加 'queued'）。queued run 不属任何进程,等 worker 认领。

**输入持久化**：agent_run 加 `enqueued_input JSONB NULL`——存 `{input, image_refs, untrusted_content}`
（NULL=同步 run，不走队列）。这是 worker 重建 graph_input 唯一不可派生的部分（其余 `system_prompt`/
`supports_vision`/`spotlight_nonce`/`max_steps` 由 `runtime.get_agent` 重建,同 `_respawn`）。认领执行后置 NULL
（瘦身;输入此后在 checkpoint/event_log 里）。

**入口分叉**：`POST /runs` body 加 `mode: "stream" | "queue"`（默认 stream,向后兼容）。`queue` 模式：
持久化 input + 建 `status=queued` 行 + 返 `202 {run_id}`（非流式）。客户端经 `GET /runs/{id}/events` 拿输出。

**RunQueueWorker**（in-process lifespan,结构同 `OrphanSweep`,每实例都跑）：
- `run_once`：`_bypass_rls` → `list_queued(limit)` → 每行 `claim_queued` CAS。
- `claim_queued` CAS：`UPDATE agent_run SET status='running', claimed_by=?, lease_until=?, heartbeat_at=?
  WHERE id=? AND status='queued'` → rowcount>0=赢家（多实例恰一）。
- 赢则 `_run_queued`：`_tenant_scope` → 读 enqueued_input → 重建 agent（thread_meta→agent_spec→
  `runtime.get_agent`,同 `_respawn`）→ 造 graph_input（共享 helper `build_run_graph_input`）→
  `run_manager.adopt`（status 已 running）→ `run_agent(graph_input=造的)` → `attach_task`。
- 心跳：worker 内 `run_agent` 已起 `_heartbeat_loop`（9.4）。queued→running 后,若该 worker 崩,OrphanSweep
  接力续跑（graph_input=None 从 checkpoint）——**两套机制天然叠加**：QueueWorker 起新 run,OrphanSweep 救崩 run。
- poison 保护：复用 `reclaim_count`；queued run 认领失败标 error（无限重排防护用 sweep 既有 cap）。

**共享 graph_input 构造**：抽 `runs.py` 的 graph_input 造法为 `orchestrator` 侧 helper
`build_run_graph_input(built, *, input_text, image_refs, untrusted_content)`,POST handler + QueueWorker 同源,
防漂移。

### 2.2 Phase 2 — 现有 worker 多副本 CAS 加固

把「list→执行」改「CAS 认领→执行」,蓝绿 exactly-once：
- **EvalWorker**：`eval_run` 加 claim CAS（`UPDATE ... SET status='running' WHERE id=? AND status='queued'`
  rowcount>0 才执行）。
- **TriggerScheduler**：cron fire 用 advisory-lock 或 `trigger_run` 的 CAS 占位（fire 前 CAS 建/认领 trigger_run
  行,已有 trigger_run 表）。
- **approval timeout sweep / token_reservation reaper**：各自认领 CAS（多为 UPDATE…WHERE status=…）。

> 不引单点选主（违 house 风格 + J-42 已否决 leader election）;CAS-exactly-once 是 house 既定模式（OrphanSweep
> 已证）。

### 2.2b Phase 2b 实施 — token_reservation reaper exactly-once（已交付）

核 `quota/sql.py` 发现真 gap **比设计预想更严重**:不只是 reaper「list→release」缺认领,而是
`reserve`/`commit`/`release` 三者本身就是 **无行锁的 read-check-write**（`session.get` → 读 `state` 判断 →
改 `token_budget_ledger`）。READ COMMITTED 下两事务都读到 `RESERVED` → 都过 `if state != reserved` 闸 →
**ledger 双退款 / 丢失更新**。reaper 每实例跑（`enable_reaper` 默认 True）→ 当前部署即触发,非仅扩展限制。

交付（house 既有 `.with_for_update()` 先例,skill/sql.py:1163）:

- **`_lock_reservation`**：transition 前 `SELECT … FOR UPDATE` 锁预约行,串行化竞争的 commit/release/expire;
  败者阻塞到赢者提交后再读到已闭合 state → 早返回不退款 → **每预约恰一次退款**。
- **ledger 算术原子化**：`reserve` 用 `INSERT … ON CONFLICT DO UPDATE SET reserved_total =
  greatest(0, reserved_total + :delta)`（SQL 内做加减,DB 行锁串行化）替代 ORM `+=`,杜绝并发 reserve 丢失更新;
  `commit`/`release`/`expire` 同款原子 UPDATE。
- **`expire_reserved(*, reservation_id, tenant_id) -> bool`** 新 store 方法（base/sql/memory）:reaper 专用,
  原子 EXPIRED 转换 + 退款,**返回是否本调用赢**。reaper 用它替代 `release(EXPIRED)`+吞 `NotFound`,
  `on_expire` 副作用（审计 emit）+ 计数 gated on won → 恰一,不再赢家输家都 emit。

验证:`test_sql_quota_store.py` 真 PG 16 并发 — expire 恰一赢（ledger 退款一次非负）/ 并发 reserve 无丢失更新 /
commit↔release 竞争退款一次;`test_quota_in_memory.py` expire_reserved 恰一 + reaper 败者跳过 hook。

> TriggerScheduler cron CAS / approval sweep 仍待后续（TriggerScheduler 当前 `enable_scheduler` 单副本闸,
> 非 live bug;reaper 是每实例真 bug 故先收）。

## 3. SSE 关系：并行共存,非替换

`mode=stream`（默认）保留现有同步 SSE 路径不变。`mode=queue` 新增异步路径。run_agent()/StreamBridge/durable
event store/RunManager 心跳/adopt **全不改**。

## 4. live 验收（★5 硬门,同 9.4）

蓝绿双实例真栈：`mode=queue` 入队一个 deepseek run → 探到 status=queued → **kill 入队实例** → 另一实例
RunQueueWorker 认领 → 执行到 `status=success`,owner=另一实例,客户端经 events 端点拿到完整输出。
验 exactly-once（只一实例认领）+ 横向扩展（加实例=更高 drain 吞吐）。

## 5. 不做 / 边界

- 不做 Celery/RabbitMQ（§ house 风格）。
- 不做跨进程 live SSE（bridge 仍 per-process;客户端经 durable events 端点重连即可,无需跨进程转发）。
- 优先级/公平调度（per-tenant 队列权重）：M1，本期 FIFO + per-tenant 并发由现有 quota 兜。
