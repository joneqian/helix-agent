# 9.4 跨进程恢复 + 9.5 托管任务队列 — 架构设计 + 选型对比（HA / failover）

> 日期：2026-06-16
> 目标：满分化域9 持久化唯二 ★2（9.4 自动 failover / 9.5 分布式任务队列）。per-user 持久 agent（always-on）核心。
> **本文是选型对比 + 方向设计**——架构主线决策，按规矩先对比（含反方）+ 收敛再编码。M1 大件，预期分阶段。

---

## 0. 现状（代码核实，三 agent 并行调研）

**底座已满分**——崩溃数据不丢：
- `agent_run` 表（migration 0032）：`status`(pending/running/success/error/timeout/interrupted/paused)，
  `ix_agent_run_thread_inflight ON (thread_id,status) WHERE status IN ('pending','running')`。
  **无 owner/lease/heartbeat 列**。`RunManager` = in-memory dict + 同步 SQL 镜像写（J-41）。
- LangGraph `AsyncPostgresSaver` checkpoint：durable，**任意进程读同库可恢复**。
- 事件日志 append-only + advisory-lock seq；H.3 `run_event` durable mirror + `GET .../events` 重放。
- **13.2 幂等 resume 刚落**（`idempotency_key`+`continuation_run_id` 随 CAS 原子写）。

**唯缺崩溃后自动编排**：
- run = 收请求进程内 `asyncio.create_task(run_agent)`，`record.task` 仅内存。
- 进程崩 → checkpoint/agent_run 在 Postgres（可恢复）**但无人接手**，`agent_run.status=running` 永久孤儿。
- **零** 孤儿检测 / 租约 / 心跳 / failover（grep confirm，仅 sandbox-supervisor 有 orphan sweep）。
- 仅人工 resume（approval gate）。

**house 风格（强约束选型）**：Postgres-first——`pg_advisory_xact_lock`（event_log seq / workspace_lock 跨副本）+
scan-CAS 队列（`eval_run` queued→running / `trigger_run` fired→retrying backoff / approval timeout sweep）遍地；
**Mini-ADR J-42 明确否决 APScheduler jobstore，改 scheduler 轮询 SQL 表**；全仓零 Celery/RabbitMQ/Kafka；
后台 worker 都是 in-process lifespan（eval/skill-evo/webhook/trigger-scheduler）或独立容器（retention/audit-backup）。

**部署**：control-plane 蓝绿（两色常驻，nginx upstream 切；staging/prod `single_instance=false`+Redis 限流）；
orchestrator in-process 库（不独立容器）；StreamBridge **进程内 in-memory**。设计文档（J-18/26/41/42）
**明确把分布式 scheduler/队列推 M1+**。

## 1. 真正的症结：SSE 流的进程内耦合

当前 POST /runs：**收请求的进程**建 agent + spawn worker + in-memory StreamBridge **同进程**流给客户端。
→ **9.4 failover 不碰这个**（只补崩溃后恢复，dispatch/streaming 模型不变）；
→ **9.5 分布式派发打破它**：别的 worker 认领 run 后，客户端 SSE 连接（实例 X）要拿 worker（实例 Y）的事件 →
需**跨进程事件流**（Redis pub/sub 或 poll durable event store）。**这是 9.5 真 XL 的点，9.4 便宜得多。**
→ 故**分阶段**：9.4 先（recovery，主代码）/ 9.5 后（distributed dispatch + 跨进程流）。

## 2. 选型对比：执行底座

| 选项 | 机制 | 优 | 反方 / 劣 | 判 |
|---|---|---|---|---|
| **A. Postgres 工作队列（SKIP LOCKED）+ 租约** | `agent_run` 加 `claimed_by`/`lease_until`/`heartbeat_at`；worker `SELECT ... WHERE status='queued' FOR UPDATE SKIP LOCKED` 认领；孤儿 sweep 重入队 | **零新基建**（Postgres 已在）；**完全契合 house 风格**（advisory lock + scan-CAS 队列遍地，J-42 已否决 APScheduler 改 SQL 轮询）；复用 durable checkpoint+event_log+13.2 幂等；团队已熟此模式 | 高吞吐时 SKIP LOCKED 轮询增 DB 负载（→ LISTEN/NOTIFY 免忙轮询缓解）；长 run 持有 claimed 行（低 churn 无妨）；agent run 是分钟级低 QPS，**非高频消息总线**故 PG 队列恰适 | ✅ **推荐** |
| **B. Celery + Redis/RabbitMQ broker** | 专用 broker + 独立 worker 池 | 成熟、专用、内建重试/超时 | **新 broker 依赖**（团队**刻意避开**：J-42 否决 jobstore，全仓零 Celery；Redis 仅作 quota cache+allkeys-lru 驱逐，**非 durable broker**，当 broker 要另起 durable Redis/RabbitMQ）；新部署+运维面；**双真相源**（broker+agent_run）；不如 A 自然复用 checkpoint/幂等机器 | ❌ 违 house 风格，新基建 |
| **C. Temporal / 持久执行引擎** | workflow 引擎内建 failover/timer/exactly-once | 持久工作流金标准 | **巨型新基建 + 范式重写**（agent 执行改 Temporal workflow）；**agent 执行已经 durable**（LangGraph checkpoint）→ Temporal **重复造**；采用成本巨大 | ❌ 过度工程（同 8.5 否 Cedar） |
| **D. 仅 recovery（租约+孤儿 sweep，不改派发）** | 保持进程内 spawn + 加租约/心跳/孤儿 sweep | 便宜收 9.4，主代码，不碰流 | **不收 9.5**（仍无分布式派发，run 钉在收请求的实例） | ＝ A 的 Phase 1 |

**结论**：**A 为目标底座，分两阶段交付**。Phase 1（= D）= 9.4 recovery；Phase 2 = 9.5 distributed dispatch。

## 3. Phase 1 — 9.4 跨进程恢复（recovery，主代码，便宜）

**不改 dispatch/streaming**：仍收请求进程内 spawn。补崩溃后恢复。

- **schema**：`agent_run` 加 `claimed_by text NULL`（实例 id）+ `lease_until timestamptz NULL` + `heartbeat_at timestamptz NULL`。migration。
- **心跳**：worker 执行中周期（如 10s）`UPDATE agent_run SET heartbeat_at=now(), lease_until=now()+interval '30s'`（复用 set_status 路径或新轻量 touch）。实例 id = 启动时生成（hostname+pid+uuid）。
- **孤儿 sweep**（新 in-process lifespan job 或并入现有 sweep，**仿 approval timeout sweep**）：扫
  `status='running' AND lease_until < now()` → 判为孤儿（owner 崩）。
- **failover 模式**（§5 决策点）：孤儿 run →
  - **(a) 保守**：标 `error`/`interrupted`（surfaces to user，安全简单），可选写事件让客户端知崩。
  - **(b) 热接力**：**自动从 checkpoint 重认领续跑**（另一实例 spawn `run_agent(graph_input=None)`，复用 13.2 幂等避重复副作用）。真 failover。
- **复用**：checkpoint durable + 13.2 幂等 + 事件日志 → 重认领续跑机制已就位，主要补「检测 + 重 spawn」。
- **范围**：1 migration + 心跳 touch + 1 sweep job +（b 选）重认领 spawn。**主代码**，无新基建。
- **收 9.4 ★2→★5**（自动 failover / 热接力）。

## 4. Phase 2 — 9.5 托管任务队列（distributed dispatch，XL，碰 SSE）

**改 dispatch 模型**：POST /runs 不在收请求进程 spawn，而**入队**（`agent_run.status=queued`）；worker 池
（control-plane 实例兼 worker，或独立 worker service）`SKIP LOCKED` 认领派发。

- **派发**：worker lifespan loop `SELECT ... WHERE status='queued' FOR UPDATE SKIP LOCKED LIMIT N` 认领→running+租约→执行。LISTEN/NOTIFY 免忙轮询（入队时 `NOTIFY`）。
- **跨进程 SSE（症结）**：认领 worker（实例 Y）≠ 客户端连接实例（X）。两解：
  - **(b1) poll durable event store**：X 轮询 H.3 `run_event` 表流给客户端（无新基建，+延迟）。
  - **(b2) Redis pub/sub**：Y publish 事件，X subscribe 实时扇出（用已部署 Redis，低延迟，+pub/sub 语义）。
  - 决策点 §5。
- **水平扩展**：run 执行不再钉收请求实例，worker 池均摊。
- **收 9.5 ★2→★5**（分布式任务队列替进程内 asyncio）。
- **XL**：dispatch 重构 + 跨进程流 + worker 池部署（运维）。

## 5. 决策（2026-06-16 用户拍板，AskUserQuestion）

1. **底座 = A（Postgres SKIP LOCKED + 租约）** ✅。否 B（Celery 新 broker 违 house 风格）/ C（Temporal 过度工程）。
2. **分期 = 先 9.4 failover** ✅（Phase 1 recovery，主代码便宜，不碰 SSE 流；9.5 之后再议）。
3. **failover 模式 = 自动热接力续跑** ✅（孤儿 run 由另一实例从 checkpoint 重认领续跑，复用 13.2 幂等；可加开关，保守标失败作 fallback）。
4. **跨进程 SSE**（Phase 2，本期不做）：留待 9.5 立项再定（b1 poll durable event store / b2 Redis pub/sub）。

## 5b. Phase 1 实施计划（9.4，已拍板）

> 目标：9.4 ★2→★5。Postgres 租约 + 心跳 + 孤儿 sweep + **自动热接力重认领续跑**。主代码、零新基建。

1. **schema**（migration）：`agent_run` 加 `claimed_by text NULL`（实例 id）+ `lease_until timestamptz NULL`
   + `heartbeat_at timestamptz NULL`。部分索引 `WHERE status='running'` 助 sweep 扫描。`RunInfo`/`RunStore`/
   `RunManager` 带这三列。
2. **实例 id + claim**：进程启动生成稳定实例 id（hostname+pid+短 uuid）。run 转 running 时 `claimed_by=自己` +
   `lease_until=now()+LEASE_TTL`（如 30s）。
3. **心跳**：worker 执行中周期（如 10s，< LEASE_TTL/2）`UPDATE agent_run SET heartbeat_at=now(),
   lease_until=now()+LEASE_TTL WHERE id=run_id`（轻量 touch，非全 set_status）。
4. **孤儿 sweep**（in-process lifespan job，仿 approval timeout sweep + trigger scheduler loop）：周期扫
   `SELECT ... WHERE status='running' AND lease_until < now() FOR UPDATE SKIP LOCKED LIMIT N`（SKIP LOCKED
   防多实例 sweep 撞）→ 每行 = 孤儿（owner 崩，心跳停，租约过期）。
5. **自动热接力**：孤儿 run → 重认领（CAS `claimed_by=自己`+新租约）→ 另一实例 `run_agent(graph_input=None)`
   从 checkpoint 续跑（checkpoint durable + 13.2 幂等 → 安全无重复副作用）。开关 `HA_AUTO_RECLAIM`（默认开；
   关则保守标 `error` reason=`orphaned`）。重认领次数上限（防 poison run 无限重跑）。
6. **可观测**：指标 `helix_run_orphan_reclaimed_total` / `helix_run_orphan_failed_total` + 审计。
7. **测试**：① 心跳更新 lease（单测）② sweep 识别孤儿（lease 过期）不误杀活 run（心跳新）③ 重认领 CAS
   竞争恰一实例赢（多 sweeper SKIP LOCKED）④ **integration 真 PG**：模拟孤儿（插 running+过期 lease 行）→
   sweep → 标重认领/失败 ⑤ 上限阻 poison。⑥ **live**：本机真栈 kill 进程 mid-run → 另一实例自动续跑完成
   （这是 9.4 ★5 的 live 验收门，同 worker 类件「CI 绿≠live 跑」教训）。
8. **运维**：sweep 作 control-plane in-process lifespan（同 eval/trigger worker），蓝绿两色都跑 sweep（SKIP
   LOCKED 安全）。无新容器/broker。
