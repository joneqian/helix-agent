## 0. 现状（基于代码核实）

**比 assessment 乐观**——06-13 后的 Stream TE-2/PI 补了多数防御：

| 维度 | 现状（code） |
|---|---|
| 调用审计 | TE-2 `_dispatch_tool`/`_emit_tool_audit`(builder.py)对**每个**工具(含 `mcp:server.tool`)emit `tool:call` 审计：arg 名(非值)/result/duration/run 归属。**MCP 已被通用审计覆盖** |
| 注入防御 | `_invoke_tool` 对 MCP 输出 `spotlight_untrusted`(当不可信);PI-2 output_screen / PI-3b action_screen 护输出/工具调用 |

→ 真剩 gap：① **MCP 专属流量维度未入审计**——通用 `tool:call` 记 arg 名+延迟，但无 **server 结构字段 / 响应字节量(exfil 体积信号) / MCP server 侧 is_error**。② **in-process 隔离无成文威胁评估**。

## 1. MCP 流量审计（code）

通用 `tool:call` 审计对 MCP 工具**补结构化流量维度**，给运维可查的 MCP 流量日志（按 server 聚合体积/错误率/exfil 信号）。

- **接入点**：`_dispatch_tool` 成功/错误路径，工具名 `startswith("mcp:")` 时构造 `mcp` 专属 details：
  - `mcp_server`：从 `mcp:<server>.<tool>` 解析（结构字段，免运维 LIKE 解析名）
  - `mcp_tool`：bare tool 名
  - `response_chars`：`len(str(outcome[0].content))`——响应体积，**exfil 体积信号**（"某 MCP 调用返回 2MB" 可疑）
  - `is_error`：MCP 结果是否错
- `_emit_tool_audit` 加可选 `extra_details: dict|None`，调用方(MCP 时)填，合入 `details`。通用路径不变。
- 隐私不破：只记 server/tool/体积/错误，**不记响应内容/arg 值**（同 TE-2 隐私规则，CodeQL clear-text-logging）。
- best-effort 不变：审计失败吞掉不挡 dispatch。

运维查：`resource_type=tool AND details.mcp_server=X` → 该 server 全流量(体积/错误/延迟/run/arg 形)。

## 2. in-process 隔离评估（威胁模型文档）

assessment 要「评估」——成文威胁模型 + 决议（HX-10 gVisor 同范式：评估+运维决议为终态）。落 `docs/adr/` 或 design doc §2：

- **边界现状**：stdio=本地子进程(独立进程,共享宿主 fs/net,operator-controlled command);remote(sse/http)=**进程内** httpx client(无子进程)。均无 gVisor/容器沙箱。
- **爆炸半径**：受损 MCP server → orchestrator 宿主权限。但 **server 配置 operator-controlled**(租户只 enable/filter,不能注入 command/url/headers)→ 租户无法引入任意 server;exfil 出口受 egress 隔离(7.7/7.8)+ 平台凭证 repr。
- **已有缓解链**(纵深):operator 准入 + allowlist + threat scan + char/tool/schema cap + 熔断 + timeout + 输出 spotlight(注入) + action_screen(越权调用) + secret repr + 本设计流量审计(检测)。
- **残留风险**:受损的 **operator 准入** server(operator 已 vet)→ 接受(信任边界=operator,非租户;与裸 Docker 沙箱信任模型一致 [[project_hx10_sandbox_isolation_decision]])。
- **M1 路径**:MCP server 进沙箱(gVisor/子进程隔离)——与 HX-10 沙箱基建并轨;记为 M1,非本期。
- **结论**:in-process 在「operator 准入 + 纵深缓解 + 流量审计检测」下**爆炸半径由信任边界(operator 非租户)界定**,接受为 M0 终态;威胁模型成文 = 评估闭环。

## 3. 测试

- `_emit_tool_audit` MCP extra_details:MCP 工具审计含 `mcp_server`/`response_chars`/`is_error`;非 MCP 工具不含(回归)。
- `_dispatch_tool` MCP 路径:解析 server、体积、错误正确入 details(成功 + 错误路径)。
- 隐私:details 不含响应内容/arg 值。
- 确定性,无模型 → CI 全覆盖(同 TE-2/3.3 档)。

## 4. 前端对接

🔶 IAM 审计页增强（SettingsAudit）——数据本可经抽屉 JSON 查（非硬 gap），但流量审计的运维价值需「可扫」非「能查」：时间线行对 `mcp:` resource_id 显示 **MCP 徽章 + inline `response_chars`**（紧凑 1.2k/3.4M），≥100k 响应 warning 高亮（exfil 体积异常一眼可见），免逐行点抽屉。`mcpTraffic`/`formatChars` 纯函数 + vitest。按 `mcp_server` 服务端过滤需 details-filter（较大，follow-up）。

## 5. 不做 / 后续

- MCP server 真沙箱(gVisor)：M1,与 HX-10 并轨。
- 响应内容入审计：隐私不可(只记体积)。
- transport 字段：server 已足够定位;transport 由 server config 派生,不冗余入每行。
- 新 `MCP_CALL` AuditAction：复用 `tool:call`+结构 details 免双审计/免破通用路径。
