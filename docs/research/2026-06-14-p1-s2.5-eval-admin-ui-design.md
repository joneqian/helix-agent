## 1. 背景与范围

S2.1a–d 已让 eval 平台后端**全闭环**：enqueue API 写 `queued` 行 → 常驻 `EvalWorker` drain
→ 跑 `run_baseline` 引擎 → 写 `eval_case_result` + summary → 状态机 `queued→running→passed/failed/error`。

但**没有 operator 前端**：`apps/admin-ui` 惯例是「每个 operator 能力配一页」，eval 是唯一只有后端
API 没有页面的能力（见 `feedback_iteration_items_need_frontend_audit`）。本工作项建该页。

**本页覆盖**：11.6（eval runs 触发 + 列表 + 单 run 结果可视化）。
**前向预留、本期不阻塞**：11.3 会话级指标 / 11.4 trace 断言 / 11.5 对抗结果——对应后端
S2.2–S2.4 未建，页面以「可选列 / 空态」前向兼容，不等它们。

## 2. 现状核对（先验证后设计，修正初版 recon）

逐项对真实后端代码核对，纠正 recon 的虚构契约：

| 事实 | 真实现状（code-evidence） | recon 误判 |
|---|---|---|
| **list 端点** | **不存在**。`api/eval_runs.py` 仅 `POST ""` / `GET /{id}` / `GET /{id}/cases` | recon 假设有 `GET /v1/eval-runs` 列表（虚构） |
| **store 列表能力** | 仅 `list_by_status_all_tenants`（跨租户，worker 用）；**无 per-tenant 列表** | — |
| **summary 形态** | 自由 dict，worker 写 `{"pass_count": N, "total": M}`（`eval_worker.py:_execute`） | recon 写 `{total_cases, passed, failed}`（虚构） |
| **case 形态** | `id/capability/case_id/passed/session_id/scores/session_metrics`（`api/eval_runs.py:_case_dict`） | 基本准 |
| **响应包装** | **raw dict**（`JSONResponse(content=_run_dict(...))`），非 `{success,data}` envelope | recon 结论正确（与 runs/curation 同为 raw，见 `feedback_envelope_vs_raw_contract_check`） |

**结论**：要建 List 页，**必须先补后端 list 端点 + store 方法**——这是 S2.5 的前置后端工作，不是纯前端。

## 3. 后端前置补口（S2.5-BE，先于前端）

### 3.1 Store 方法（`persistence/eval/{base,memory,sql}.py`）

新增 per-tenant 列表（仿 `RunStore` 列表分页范式）：

```python
async def list_runs(
    self, *, tenant_id: UUID, status: EvalRunStatus | None = None,
    limit: int = 50, offset: int = 0,
) -> tuple[list[EvalRunRecord], int]:
    """按 tenant 列出 eval runs（可选 status 过滤），返回 (页, 总数)。
    created_at DESC 排序。SQL 实现走 RLS（tenant_id 由 current_setting 注入）。"""
```

- InMemory：内存过滤 + 排序 + 切片。
- SQL：`SELECT ... WHERE (:status IS NULL OR status=:status) ORDER BY created_at DESC LIMIT/OFFSET` + `COUNT(*)`；
  tenant 隔离靠 FORCE-RLS（不显式拼 tenant_id，与现有 store 一致）。

### 3.2 List 端点（`api/eval_runs.py`）

```
GET /v1/eval-runs?status=<queued|running|passed|failed|error>&limit=&offset=&tenant_id=<uuid|*>
```

- 默认 home-tenant（`request.state.tenant_id`）。
- `tenant_id=*` 跨租户：仿 `runs.py` 的 `cross_tenant_query_enabled` 守卫——仅 `system_admin`
  放行，否则 403；跨租户读 FORCE-RLS 表要 `SET LOCAL ROLE audit_reader`
  （见 `reference_cross_tenant_read_force_rls_set_role`）。
- 返回 raw：`{"items": [_run_dict...], "total": N, "cross_tenant": bool}`（与 runs list 同形）。

> 注：保持 raw（不套 envelope），与同模块 POST/GET 及 runs/curation 一致。前端 SDK 直接解 raw，
> **不走 `getJson()`**（envelope-vs-raw 契约对账）。

### 3.3 测试

- store：`list_runs` 单测（status 过滤 / 分页 / DESC 序 / RLS 隔离）。
- API：list 端点 happy path + 空 + status 过滤 + 跨租户 403（非 admin）+ 跨租户 200（admin）。

## 5. 页面设计

### 5.1 EvalRunsList（列表 + 触发）

布局对齐 `RunsList.tsx`（philosophy §3.2 中等密度，表格行高 36–40）：

- **PageHeader**：标题「Eval Runs」+ 右侧操作区：
  - `Enqueue` 主按钮 → `POST {suite:"m0_baseline"}`（suite 客户端 allowlist，与后端 `_ALLOWED_SUITES` 一致）。
    触发后乐观刷新列表（状态可见，philosophy §3.4——不静默成功）。
  - status 过滤 `Select`（all/queued/running/passed/failed/error）。
  - `Refresh` 按钮（手动刷新）。
  - 跨租户时显示 `cross-tenant` Tag（多租户安全感 §3.5；scope 永远可见）。
- **Table** 列：Run ID（截断 8 + Tooltip 全 id + `code` 字体）· **Status badge** · Suite ·
  Summary（`{pass_count}/{total}`，缺则 `—`）· Created（本地时间）。行点击 → detail。
- **空态**：home vs 跨租户两种文案（`Empty`）。
- **状态轮询**：列表内有 `queued/running` 行时，轻量轮询刷新（复用 runs 的 `useStatusPolling` 范式），
  让 worker 推进可见；无活跃行则停轮询。

### 5.2 EvalRunDetail（单 run 结果）

布局对齐 `RunDetail.tsx`：

- **PageHeader**：`Eval Run <id 前 12>…` + 返回「Eval Runs」+ status badge + Refresh。
- **Metadata Card**（`<dl>` grid）：run id（全）· status · suite · triggered_by · created/started/finished ·
  summary（`pass_count/total`）。
- **Cases Card**（Table）：Case ID · Capability · **Result badge**（passed/failed）· Scores（紧凑 KV，
  如 `judge_mean=0.9 recall@5=0.8`）。
- **前向预留列**（本期空态、后端建后填）：
  - `Session metrics`（11.3 / S2.2）：`session_metrics` 非空时显示 resolution_rate 等。
  - `Trace`（11.4 / S2.4）：将来按 trace_id deep-link Langfuse（复用 RunDetail 的 `TraceToolbar` 范式）。
  - `Adversarial`（11.5 / S2.3）：对抗 case 的 expected_refusal 判定列。
  > 这些列在 DTO 字段缺省时不渲染，**不阻塞本期**。

### 5.3 Status badge — 色 + 形 + 字（philosophy §3.6，三者齐两）

五态映射（Antd Tag color + 文本，截黑白仍可辨）：

| status | color | 文本 | 语义 |
|---|---|---|---|
| queued | default（灰） | queued | 待 drain |
| running | processing（蓝转） | running | 执行中 |
| passed | success（绿✓） | passed | 全 case 通过 |
| failed | error（红） | failed | 至少一 case 失败 |
| error | error（红） | error | 引擎/系统错（非 case 失败） |

`failed` 与 `error` 同红，靠**文本**区分（符合「不靠颜色单独传递语义」）。

## 6. SDK 契约（`src/api/eval_runs.ts`）

raw-dict，全部经 `apiClient` 直解 `.data`，**不用 `getJson`**：

```ts
type EvalRunStatus = "queued"|"running"|"passed"|"failed"|"error";
interface EvalRunRecord { id; suite; status; triggered_by;
  summary: Record<string, unknown> | null;          // 渲染时读 pass_count/total
  created_at; started_at: string|null; finished_at: string|null; }
interface EvalCaseResult { id; capability; case_id; passed: boolean;
  session_id: string|null; scores: Record<string, number>|null;
  session_metrics: Record<string, unknown>|null; }     // 11.3 预留

enqueueEvalRun(body:{suite:string}): Promise<EvalRunRecord>       // POST → 202 raw
listEvalRuns(p:{tenantScope?,status?,limit?,offset?}): Promise<{items;total;cross_tenant}>  // GET（§3.2 新端点）
getEvalRun(runId): Promise<EvalRunRecord>                         // GET /{id} raw
getEvalRunCases(runId): Promise<{cases: EvalCaseResult[]}>        // GET /{id}/cases raw
```

`summary` 故意保持 `Record<string,unknown>`（后端是自由 dict），渲染层 `summary?.pass_count` 取值，
不在类型上锁死后端可能演进的 summary 形。

## 10. 风险 / 坑

- **antd message** 必经 `App.useApp()`（不顶层 import）。
- **跨租户 403**：非 system_admin 不给 `*` 选项；后端守卫兜底（§3.2）。
- **轮询节流**：仅活跃态轮询，避免 worker 5min 间隔下的无效高频刷新。
- **summary 自由 dict**：前端容错读 key，缺字段显 `—`，不假设固定形。
- **前向列**：11.3/11.4/11.5 字段缺省即不渲染，本期不引这些列的强依赖。
