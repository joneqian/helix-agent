# 13.2 并发 resume 幂等 — 设计（确定性恢复）

> 日期：2026-06-16
> 目标：满分化 13.2 多用户确定性恢复 ★3→★5。assessment 原文：机制设计在（idempotency_key+replay /
> token_reservation 状态机），**并发 resume race condition 未详述**；★5 需「补悲观锁 + 并发 resume race 测试」。
> 纯代码确定性，无模型、无运维依赖。

---

## 0. 现状（基于代码核实）

- resume = J.8 审批续跑：`POST /v1/sessions/{tid}/runs/{rid}/resume`（流式）+ 批量 `POST /v1/approvals:decide`，
  共用 `api/runs.py:apply_approval_decision`。
- **并发守卫已在**：`mark_decided` 是**原子条件 UPDATE**（SQL `UPDATE ... WHERE status='pending'` →
  rowcount>0 才赢；in-memory check-then-set 无 await 间隙，asyncio 协作调度下原子）。两个并发 resume →
  DB 行锁序列化，恰一个赢、其余 rowcount=0 → `decided=False` → **409**。**exactly-once 决策已正确**。
- **token_reservation 未与 resume 路径耦合**（assessment 引用是设计文档，非实际接线）。
- 续跑 worker：仅 CAS 赢家越过 `if not decided: 409`（loser 在任何副作用前退出）→ **恰一个续跑 worker**。

→ **「补悲观锁」是次优**：原子 CAS 优于悲观锁（无锁等待、无死锁面）。真 gap=① 并发 race **未证**（无测）
② **重试不幂等**：网络重试/并发同请求在 CAS 后拿 409，**客户端拿不到续跑 run_id**，无法确定性恢复。
任务名「并发 resume **幂等**」点出正题=idempotency。

## 1. 设计 — idempotency_key 确定性恢复

approval 行原子记录 `idempotency_key` + `continuation_run_id`（随 CAS 一次写），重复 decide 同 key →
**幂等返回同 `continuation_run_id`**（非 409 / 非二次 spawn）。

- **schema**：`agent_approval` 加 `idempotency_key text NULL` + `continuation_run_id uuid NULL`。migration `0080`。
- **mark_decided**：加 `idempotency_key` + `continuation_run_id` 入参，**随 CAS UPDATE 原子写**（赢家才落）。
- **apply_approval_decision 重排**：`continuation_run_id` 在 `mark_decided` **之前**生成并传入（原子绑定）；
  返回 `(run_record, continuation_run_id, replayed: bool)`。
- **幂等路径**：
  1. `get_by_run` 显示**已决** + `approval.idempotency_key == 请求 key`(非空) + 有 `continuation_run_id`
     → 幂等成功（`replayed=True`，**不 spawn**）。
  2. CAS 赢 → spawn 续跑 worker（`replayed=False`）。
  3. CAS 输（并发 race）→ 重读；同 key 命中 → 幂等返回赢家 `continuation_run_id`；否则 409。
  - 无 key（默认）→ 同今日：赢家 spawn，其余 409。**向后兼容**。
- **流式端点重放**：`resume_run` 在 `replayed=True` 时返 `JSONResponse{run_id, idempotent_replay:true}`
  + `X-Helix-Run-Id`（端点签名已是 `StreamingResponse | JSONResponse`）；客户端经现有
  `GET /{tid}/runs/{continuation_run_id}/events?since_seq=`（H.3 durable mirror）重附事件流。
  批量 decide 端点本就返 run_id → 天然幂等。

### 语义表

| 场景 | key | 结果 |
|---|---|---|
| 并发 resume，无 key | — | 恰一赢 spawn，余 409（不变） |
| 并发 resume，同 key | k | 恰一赢 spawn，余幂等返同 continuation run（尽力，read race 退化 409） |
| 决策后重试，同 key | k | 幂等返同 continuation run（**确定性恢复**，主用例） |
| 已决重试，异 key/无 key | — | 409（真冲突） |

## 2. 落地分层（单分支 `stream-j8/13.2-resume-idempotency`，1 PR）

1. **protocol**：`ApprovalRecord` 加 `idempotency_key: str|None` + `continuation_run_id: UUID|None`
2. **持久化**：`AgentApprovalRow` 2 列 + migration `0080_approval_idempotency`（≤32）；`mark_decided` 加参
   原子写（memory+sql+base）
3. **control-plane**：`ResumeRequest` + 批量 decide body 加 `idempotency_key: str|None`；
   `apply_approval_decision` 重排+幂等路径；`resume_run` replay 走 JSON
4. **测试**：见 §3

## 3. 测试（确定性，CI 全覆盖）

- **并发 race（exactly-once，assessment 点名）**：
  - 单测（in-memory）：N 个并发 `mark_decided` 同 run → 恰 1 True。
  - **集成（真 PG，关键）**：N 个并发 `mark_decided` → DB 恰 1 行更新（证 DB 行锁原子，非仅 asyncio）。
  - 端点：2 个并发 resume → 1×stream/200 + 1×409，恰 1 续跑 worker。
- **幂等重试**：同 key 重复 decide → 同 `continuation_run_id`，无二次 spawn；流式端点 replay 返 JSON。
- **冲突**：已决 + 异 key → 409。
- 无 live（同 15.3/3.3 档，确定性）。

## 4. 不做 / 后续

- 悲观锁：CAS 已最优，不引入。
- 崩溃恢复（决策后 worker 未起）：=9.4 failover（J-41），本期不含。
- token_reservation 幂等：未与 resume 耦合，无需动。
- 并发同 key 的强一致幂等（read race 退化 409）：主用例是顺序重试已确定性；并发同 key 罕见，尽力即可。
