> 两个具名短板补到 ★5。本设计经参考实现（hermes-agent / openclaw / deer-flow）+ 2026 业界资料

## 1. 现状与 gap

五大编排模式现 ★4。四个已扎实：
- **Routing** — `orchestrator/llm/router.py`
- **Parallelization** — `builder.py` `plan_stages()` + `asyncio.gather`（`SubAgentTool.is_parallel_safe=True` → 兄弟委派并发）
- **Prompt-chaining** — `plan_execute` 图
- **Evaluator-Optimizer** — `graph_builder/reflect.py` accept/revise 闭环

评估列两个具名短板，但**核现状后 gap-① 已具备**（staleness），只剩 gap-②：

### gap-① Evaluator-Optimizer 评判用同一模型 —— ⚠️ 评估表过期，能力早已具备
评估写「评判用同一模型」描述的是**默认行为**，但漏看 **Stream J.11 `routing` 早已支持给 reflection 步骤类指定独立模型**：
- `RouteRule(when="reflection", model=ModelSpec)`（agent_spec.py:336）—— 独立模型 + 自带 fallback 链
- `build_step_routers:1400` 对 `when=reflection` 建独立 router；`make_reflect_node(routers.reflection)`（agent_factory:528）运行时真用它
- 测试 `test_build_step_routers_routes_reflection_independently`（test_agent_factory.py:557）已证：主模型 anthropic、reflection 路由到 openai/gpt-4o-mini
- **UI 也能配**：admin-ui `ManifestEditor` 是 schema 驱动（Form 视图自动渲染 `routing` 段 + YAML 视图直接写）

→ 独立评判者**后端+UI 都已 ★5，零必做工作**。**刻意不复用 platform_judge**（安全审查快模型，弱模型评强模型反变弱）；默认不配 = 复用主模型（opt-in 选独立，正确设计）。
→ **唯一 UX 缺口**：Form 视图里 routing 是通用「rules 数组」自动控件，无「评判者模型」专门起名+说明。按「Admin UI 必须产品级」做一个友好控件（Phase 1-FE，UX 打磨，非能力/评分项）。

### gap-② Orchestrator-Worker「仅基础」= 静态名单，非动态创建
现状 `spec.subagents`（agent_spec.py:710 `SubAgentSpec`）是 **manifest 部署时写死的一组 agent_ref**。`SubAgentTool`（tools/subagent.py）把每个声明项包成命名工具，父 LLM 运行时只能在**预声明的几个**里选。

**调研结论：2026 主流是动态创建临时 worker，不是从静态名单选。**

| 参考实现 | worker 来源 |
|---|---|
| hermes-agent | 完全动态：`_build_child_agent` 运行时生成 id + 现场构建 child system prompt（按 goal+role）+ 动态工具集，零池化 |
| openclaw | 动态 session：每次 spawn 新 UUID + 运行时构建 prompt + 按深度定角色 |
| Claude Code（Task 工具）| 动态 ephemeral：父现场 spawn，fresh session，完成即弃 |
| Anthropic Research 系统 | 动态：orchestrator 现场生成 3-5 个 subagent 任务描述，临时；官方原文强调 "dynamically breaks down... delegates" |
| deer-flow | 静态名单（builtin + config.yaml 选名）—— 老式 |
| LangGraph supervisor / OpenAI handoff | 静态池 —— 但都在加动态（Deep Agents） |
| 学术 AgentSpawn | 动态 spawning 完成率比静态 **+34%** |

生产多租户最佳实践 = **动态 spawn（应用层）+ 静态隔离基建（沙箱/配额，基础层）**。helix 静态基建全有（gVisor/HX-10 沙箱、per-tenant 配额、`MAX_SUBAGENT_DEPTH=3`、subagent 运行时），缺的就是「动态 spawn 临时 worker」这一层。

## 2. 不做什么（明确排除）

- **不做独立 coordinator 节点 + synthesizer 节点**（Explore 初版 630 行方案）。理由：父 agent **本就是 orchestrator**——它运行时决定调哪个 worker（委派）+ 整合 tool 结果产最终答案（综合）。Anthropic / Claude Code / hermes / openclaw **都无独立 coordinator 节点**，父 LLM 即协调者。再加独立节点 = 重复父职责 + 多 LLM 调用 + 延迟成本，换边际能力。综合也无需专门节点（父 LLM 读 worker 结果即综合，与 SubAgentTool 现状一致）。
- **不做持久 worker 池**（预热 N 个 worker 复用）。ephemeral 跑完即弃，符合 Claude Code / Anthropic Research 形态；池化是基础设施层（沙箱）的事，已由 HX-10 覆盖。
- **不改静态 `spec.subagents`**。动态 spawn 与静态命名委派**并存**：静态适合「已知角色、想给 LLM 强选择信号」（命名工具），动态适合「子任务不可预知、需现场生成角色」。两者共用同一 child run 机制。

## 3. gap-① 独立评判者：后端+UI 已具备，只做 UX 友好控件

### 3.1 后端已完整（零工作）
独立评判者经现有 Stream J.11 `routing` 配置：manifest 写 `routing.rules[{when: reflection, model: <独立 ModelSpec>}]` → reflect 节点路由到独立模型（自带 fallback + 金库凭证解析，Y-2 路径）。code + wired + tested（见 §1 gap-① 证据）。**不加 `ReflectionSpec.model`——会与 routing 冗余成两种配法。**

### 3.2 UX 缺口 + 友好控件（Phase 1-FE）
现状：admin-ui Form 视图从 schema 自动渲染 `routing` 为通用「rules 数组」控件（when 下拉 + 嵌套 model）——能用但不直观，用户不知道「这是给反思配独立评判者」。

**做一个友好控件**（manifest 编辑器 Form 视图增强）：在 `reflection` 配置区旁加「评判者模型」可选控件——
- 标签 + 说明：「反思评判者模型（留空=用 agent 自己的模型；建议选同级/更强/不同厂商以换视角）」
- 选了模型 → 编辑器在 manifest 里**写成 `routing.rules[{when: reflection, model: ...}]`**（即友好控件是 routing 的「reflection 专门视图」，底层仍是 routing，不引入新字段）
- 复用现有 model picker（catalog.ts 已有 provider/model 选择器）

**关键**：控件只是 routing 的 UX 投影，**不改协议、不改后端**。读时：manifest 里有 `when=reflection` 规则 → 控件回显该 model；无 → 控件空。

### 3.3 量级（纯前端 ~120 行）
manifest-editor 加 reflection-evaluator 控件 + 与 routing 互转（读/写）+ i18n 双语 + Storybook + Playwright。无后端/协议改动。SE-8 接线点：仅 manifest-editor 组件内 + i18n（不新增页/路由/SDK）。

## 4. gap-② 设计：动态 worker spawn

### 4.1 核心思路
新增 `spawn_worker` 工具（与 `SubAgentTool` 并列），父 LLM 现场调用：给一段 `task` 描述 + 可选 `focus`/角色提示 → orchestrator 用**生成的临时 spec** 建一个 ephemeral worker → 跑到完成 → 返回最终答案 → 父综合。复用 `SubAgentTool` 已有的全部 child run 机制。

### 4.2 复用现有机制（~90%）
`SubAgentTool.call()`（subagent.py:174）已有：深度 cap（`child_depth`/`MAX_SUBAGENT_DEPTH`）、共享 `CancellationToken`、`deadline_at` 传播、L7 轨迹录制（`_dispatch_trajectory`）、`is_parallel_safe`（兄弟并发）、`_final_answer` 提取、`SubAgentInvocation` 上报、max_steps 部分结果处理。**动态 spawn 全部复用。**

唯一差异：child 从哪来。
- 静态 `SubAgentTool`：`builder(tenant_id, name, version, depth)` → `spec_store.get` → `build_agent(record.spec)`（subagent_runtime.py:91-125）。
- 动态 `SpawnWorkerTool`：`worker_builder(tenant_id, instructions, role, depth)` → **生成临时 spec**（worker 模板 + 注入 instructions/role）→ `build_agent(generated_spec)`，**跳过 spec_store**。

### 4.3 新增组件

**(a) Worker 基座模板（protocol 或 control-plane）**
ephemeral worker 的最小 `AgentSpec` 模板（非用户部署，运行时合成）：
- `model`：**直接继承父 spec 的 model，不可配**（用户拍板——少一个配置面；worker 与父同能力）
- `system_prompt`：`= 生成的 instructions`（父 LLM 现场写的角色+任务）
- `sandbox`：**继承父的隔离配置**（同租户、同 gVisor、同 egress 策略）—— 安全边界不放松
- `tools`：按 `role` 从父声明的 `allowed_toolsets` 取子集（默认继承父工具集 ∩ 安全白名单）
- `memory/triggers/skills/subagents`：**空**（ephemeral 无状态；worker 不能再静态委派，但若 depth 未到顶可再 spawn）
- `reflection`：可选继承
- `workflow`：react，`max_iterations` 受全局 `worker_max_iterations` 上限约束

**(b) `make_worker_agent_builder`（control-plane，subagent_runtime.py 旁）**
仿 `make_child_agent_builder`，但 `_build(tenant_id, instructions, role, depth)` 不查 spec_store，而是 `_synthesize_worker_spec(parent_spec, instructions, role)` → `build_agent(...)`。复用 provider_key_resolver / sandbox / checkpointer / depth 全套。缓存**不按 instructions 缓存**（每次现场生成，ephemeral）；但 worker 模板的解析（模型/工具）可缓存。

**(c) `SpawnWorkerTool`（orchestrator，tools/spawn_worker.py）**
- `ToolSpec`：name=`spawn_worker`，params=`{task: str (必填), focus?: str (角色/专长提示)}`，`is_parallel_safe=True`（多 worker 并发，同 SubAgentTool）。
- `call()`：把 `SubAgentTool.call()` 的 child run 主体抽成共享 helper（`_run_child_agent`），两个工具都调。差异仅 child 来源（builder vs worker_builder）+ instructions 拼装。
- 父 LLM 现场决定 `task`（含它想让 worker 扮演什么角色），`focus` 辅助生成 system_prompt。

**(d) 配置分两层：全局边界（settings）+ per-agent 开关（manifest）**

依据「业界并发限制优先全局」+ helix 现状（`MAX_SUBAGENT_DEPTH=3` 即写死全局常量）+ 多租户安全（硬上限保护共享沙箱/配额，不让 manifest 作者自己调高）：

**全局边界（control-plane settings.py）** —— 运维设一次，所有 agent 共用，是硬上限：
```python
enable_dynamic_workers: bool = True                    # 平台主开关，默认开
dynamic_worker_max_concurrent: int = Field(default=3)  # 同时并发上限（行业铁打 3）
dynamic_worker_max_per_run: int = Field(default=16)    # 单 run 累计 spawn 上限（runaway 安全网）
dynamic_worker_max_iterations: int = Field(default=8)  # 每 worker 步数上限
dynamic_worker_allowed_toolsets: list[str] = []        # 全局工具白名单（空=继承父∩内置安全集）
```

**per-agent（protocol/agent_spec.py，manifest）** —— 只一个 opt-out 开关，**不给数值**（避免突破全局安全上限 + 保持简单）：
```python
class DynamicWorkersSpec(BaseModel):
    model_config = ConfigDict(extra="forbid")
    enabled: bool = True   # 默认继承平台（开）；某 agent 想关就 false
```
挂在 `AgentSpec.spec` 下与 `subagents` 并列。`spec.dynamic_workers` 整块省略 = 用平台默认（开）。`enabled=false` → 不注册 `spawn_worker` 工具。

**数值边界全局兜**，per-agent 不可调高——并发/总数/步数/工具白名单都读 settings。per-tenant 公平性由**现有配额引擎**管（worker run reserve token），不在此重复。

**默认开的取舍**：每个顶层 agent（depth 0）默认带 `spawn_worker` 工具（多一点 prompt token + LLM 可能用它）。用户拍板默认开——契合 per-user 持久 agent「能调度 worker」的形态；简单 agent 可 manifest opt-out。

### 4.4 安全边界（生产级，非放松）
| 维度 | 机制 | 复用 |
|---|---|---|
| 递归失控 | worker 在 `child_depth` 建，到 `MAX_SUBAGENT_DEPTH=3` 不再带 spawn 工具 | 现有深度 cap |
| 成本失控 | per-tenant 配额（worker run reserve token）+ 全局 `dynamic_worker_max_per_run` + `_max_concurrent` + `_max_iterations` + 父 `deadline_at` 传播（整棵树同一 wall-clock） | 现有配额 + deadline + 全局 settings |
| 沙箱逃逸 | worker 继承父 sandbox（gVisor/egress/fs），不放松 | HX-10 沙箱 |
| 跨租户 | worker spec `tenant_id` = 父租户，工具/凭证全租户作用域 | 现有租户隔离 |
| 工具越权 | worker 工具 = 父工具集 ∩ 全局 `dynamic_worker_allowed_toolsets`（空=内置安全集）；默认不给比父更多 | 工具注册 |
| 提示注入 | worker 的 task 来自父 LLM（不可信通道经 spotlight 已包裹）；worker 自身 DefenseSpec 继承父 | PI-1/2 |

`dynamic_worker_max_per_run` 计数累加在 run 级 state（`AgentState`），超限 `spawn_worker` 返回工具错误（非崩溃），父可据此收敛。所有数值读全局 settings（§4.3d），per-agent 不可调高。

### 4.5 综合策略（无新节点）
worker 结果 = `ToolResult`（final answer）→ 变 `ToolMessage` 追加 messages → 父下一步 LLM 调用**隐式综合**（与 SubAgentTool 现状一致，与 Claude Code / hermes 一致）。多 worker 并发结果按 `plan_stages` 并行收集后一起回父。**不加 synthesizer 节点**（§2）。

### 4.6 可观测
worker run 复用 `_dispatch_trajectory`（L7 轨迹，J.13 eval 可 replay）+ `SubAgentInvocation` 上报（metadata 标 `dynamic=true`、role、生成的 instructions 摘要）。新增指标 `helix_dynamic_worker_spawned_total{role}` / `_blocked_total{reason}`（超限/越权）。admin-ui 维度：RunDetail 已有 subagent 调用树展示，dynamic worker 复用同展示（标记 ephemeral）。

### 4.7 量级 ~350 行（后端）
DynamicWorkersSpec(只 enabled,10) + 全局 settings(15) + worker 模板合成(60) + make_worker_agent_builder(70) + SpawnWorkerTool + 抽共享 _run_child_agent(110) + agent_factory/assembly 接线(45) + 指标(20) + 测试(§6)。admin-ui 平台 section 另计（§4.8，按需 Phase 2）。

## 6. 测试计划（TDD，≥80%）

**gap-①（后端已测，只测新 FE 控件）**：
- 后端独立评判者已有覆盖（`test_build_step_routers_routes_reflection_independently` 等），无新增。
- FE 单测（vitest）：评判者控件选模型 → manifest 写出 `routing.rules[{when:reflection,model}]`；清空 → 移除该规则；已有 `when=reflection` 规则 → 控件回显。
- FE 单测：与其他 routing 规则（planning）共存不互踩。
- Playwright：编辑器里选评判者模型 → 切 YAML 视图核 routing 正确。

**gap-②**：
- 单测：`_synthesize_worker_spec` 从父 spec + instructions 生成正确 worker spec（继承 sandbox/tenant、工具白名单 ∩、空 memory）。
- 单测：`SpawnWorkerTool.call` 复用 child run（fake worker_builder + fake graph，断言 final answer 提取 / 深度传播 / 取消传播 / max_steps 部分结果）。
- 单测：边界——`max_workers_per_run` 超限返工具错误非崩溃；depth 到顶不注册 spawn 工具；越权 toolset 被裁。
- 装配测：`dynamic_workers.enabled=false` → 无 spawn_worker 工具（零行为变化）。
- 集成测（CI 无模型，用 fake LLM）：父 spawn 2 worker 并发 → 收集 → 综合。
- **live E2E（真栈 deepseek）**：父 agent 真 spawn worker 干子任务、真综合（照本轮 live-smoke 起栈链，捞静态测漏的接线/凭证休眠）。

## 7. 分期

- **Phase 1（纯前端 UX）**：gap-① 评判者模型友好控件（§3.2）—— manifest 编辑器加「反思评判者模型」控件（routing 的 reflection 专门视图，不改后端/协议）+ i18n + Storybook + Playwright。**后端独立评判者已具备，本 phase 只补 UX。**
- **Phase 2（后端）**：gap-②（动态 spawn）—— 抽共享 child run helper → worker 模板 + builder → SpawnWorkerTool → 全局 settings 边界 → manifest opt-out 开关 → 后端测试 → live E2E。
- **Phase 2-FE**：admin-ui（§4.8）—— manifest schema 加 `dynamic_workers` + RunDetail worker 树增强（标 ephemeral/role）+ 平台边界 section（对标 embedding-config，SE-8 全套接线）+ i18n。**显式子项，不漏。**

各 phase 独立 PR。Phase 1（FE）/ Phase 2（后端）互不依赖；Phase 2-FE 依赖 Phase 2 后端。1.3 ★4→★5 由 Phase 2 决定（gap-②），Phase 1 是 UX 打磨。

## 8. 评分依据

★5 = 代码存在 + 接线生效 + 测试覆盖 + （worker 类）live E2E 实证。
